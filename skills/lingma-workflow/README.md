# Lingma Workflow Skill - 使用指南

## ⚖️ 铁律约束（必须遵守）

**使用本技能前，必须了解并遵守以下三条“法律”：**

### 🔴 铁律 1：入口文件验证（智能检测与自举）

**优先使用 `.lingma/LINGMA.md`，支持自动创建和多入口识别**

#### 核心原则

- ✅ **默认行为**：自动使用 `.lingma/LINGMA.md` 作为标准入口
- 🤖 **自动创建**：如果不存在，提供模板生成或引导创建
- 🔍 **智能识别**：允许在特定条件下识别替代入口
- 🚀 **自举能力**：确保工作流能够从最小配置启动

#### 处理流程

**场景 1：标准入口存在**
```
.lingma/LINGMA.md ✅ 存在
→ 直接使用
→ 无提示，正常执行
```

**场景 2：标准入口不存在，有替代品**
```
.lingma/LINGMA.md ❌ 不存在
↓ 扫描替代入口
发现：.lingma/workflow/README.md ✅
↓ 提示用户
⚠️ 未找到标准入口，但发现了可能的入口文件：
   .lingma/workflow/README.md
   
请选择：
1. ✅ 使用此文件作为临时入口（推荐）
2. 📝 立即创建标准入口
3. 📋 从模板生成
```

**场景 3：标准入口不存在，无替代品**
```
.lingma/LINGMA.md ❌ 不存在
↓ 扫描替代入口
未发现任何可用入口 ❌
↓ 启动自举流程
🤖 正在为您创建入口文件...

选择模板类型：
1. 📄 基础模板（最小化配置，50 行）
2. 📊 完整模板（包含所有推荐章节，200 行）
3. 🎯 智能生成（分析项目后定制）

→ 创建 .lingma/LINGMA.md ✅
→ 记录到 bootstrap.log
→ 继续执行
```

#### 示例用法

```bash
# ✅ 正确（标准情况）
/lingma-workflow
# 自动使用 .lingma/LINGMA.md

# ✅ 正确（入口文件不存在时）
/lingma-workflow
# 触发：🤖 未找到入口文件，正在启动自举流程...
# → 提供选项 → 创建 → 继续执行

# ❌ 错误（会被拒绝）
/lingma-workflow --entry src/workflow/entry.md
# 除非 .lingma/LINGMA.md 确实不存在且用户明确选择
```

#### 支持的入口文件类型

| 入口路径 | 优先级 | 置信度 | 说明 |
|---------|--------|--------|------|
| `.lingma/LINGMA.md` | ⭐⭐⭐⭐⭐ | 1.0 | 标准入口（首选） |
| `.lingma/workflow/README.md` | ⭐⭐⭐⭐ | 0.8 | 如果有明确的工作流说明 |
| `LINGMA.md` (根目录) | ⭐⭐⭐ | 0.7 | 历史遗留项目 |
| `.lingma/workflow/index.md` | ⭐⭐ | 0.6 | 索引式入口 |

#### 与旧版本的区别

| 特性 | 旧版本（严格模式） | 新版本（智能模式） |
|------|-------------------|-------------------|
| 入口不存在时 | ❌ 直接拒绝 | 🤖 自动创建 |
| 替代入口 | ❌ 不允许 | ✅ 智能识别 |
| 用户体验 | 🔴 中断 | 🟢 流畅 |
| 自举能力 | ❌ 无 | ✅ 完整 |
| 灵活性 | 🔴 低 | 🟢 高 |

### 🔴 铁律 2：路径约束

**工作流生成的路径必须在工作空间根目录下的 `.lingma/workflow` 内**

- ✅ 允许的路径：`.lingma/workflow/任意文件.md`
- ❌ 禁止的路径：`src/workflow/xxx.md`、`docs/workflow/xxx.md`
- 🛑 后果：如果检测到目标路径超出范围，技能将拒绝执行

**示例：**
```markdown
✅ 正确：
- .lingma/workflow/README.md
- .lingma/workflow/project_guidelines.md
- .lingma/workflow/sub-folder/xxx.md

❌ 错误（会被拒绝）：
- src/workflow/process.md
- docs/workflow/spec.md
- .lingma/other/workflow/xxx.md
```

### 🔴 铁律 3：Guidelines 优先原则

**`project_guidelines.md` 是最重要的核心文档，必须谨慎维护**

创建或更新 Guidelines 时，**必须包含以下 7 个核心章节**：

1. **智能体角色定义**（Role）
2. **优先事项**（What Lingma Should Prefer）
3. **避免事项**（What Lingma Should Avoid）
4. **响应规则**（Response Rules）
5. **术语规范**（Current Canonical Terms）
6. **交付标准**（Deliverable Standard）
7. **Workflow 完善建议**（Suggested Next Deliverables）

- 🛑 后果：如果 Guidelines 不完整，技能会要求先补全再执行

## 技能概述

`lingma-workflow` 技能确保智能体在处理项目任务时，自动遵循项目的标准化工作流和文档规范。

## 核心能力

### 0. 铁律验证（最高优先级）

在任何操作前，智能体会自动验证：

```markdown
验证清单：
□ 铁律 1：入口文件是否为 .lingma/LINGMA.md？
□ 铁律 2：所有操作路径是否在 .lingma/workflow 内？
□ 铁律 3：如涉及 Guidelines，是否已包含全部 7 个核心章节？
```

**任何一项验证失败 → 立即停止并返回错误信息**

### 1. 自动文档对齐

智能体会自动读取并参考以下关键文档：

- `.lingma/LINGMA.md` - 工作流入口文件
- `.lingma/workflow/README.md` - 目录说明
- `.lingma/workflow/project_guidelines.md` - 协作规范
- `.lingma/workflow/WORKFLOW.md` - 执行流程
- `.lingma/workflow/task_breakdown.md` - 任务分解与进度

### 2. 阶段一致性检查

智能体会根据项目当前阶段（见 `project_status.md`）调整建议，避免：
- 过度设计（超前于当前阶段）
- 滞后行为（重复已完成的工作）
- 范围发散（在 M1 前引入不必要的功能）

### 3. 标准化输出

所有响应都包含以下结构：

```markdown
## 事实依据
说明哪些内容来自现有文档确认

## 建议方案
基于工作流规范的可执行方案

## 待确认项
列出需要补充或确认的信息

## 进度追踪
更新任务分解文档中的状态（如适用）
```

## 触发场景

### 自动触发

当用户请求中包含以下关键词或场景时，技能会自动触发：

- "工作流"、"流程"、"规范"
- "按照项目标准"
- "符合项目规范"
- "下一步做什么"
- "项目进展"、"任务进度"
- "创建新 X"（X 为功能、组件、文档等）
- "检查是否符合"

### 手动触发

用户可以通过以下方式手动触发技能：

```
使用 /lingma-workflow 命令
或明确要求"请按照项目工作流处理"
```

## 使用示例

### 示例 1：新功能开发

**用户**：我们需要添加一个新的订单导出功能

**技能执行过程**：
1. 读取 `task_breakdown.md` 查看任务优先级
2. 检查 `project_guidelines.md` 确认交付标准
3. 分析现有订单模块的代码模式
4. 输出完整的实现计划

**预期输出**：
```markdown
## 事实依据
- 项目当前处于订单管理模块完善阶段（project_status.md）
- 已有订单列表和详情组件（代码库确认）

## 建议方案
1. 新增订单导出按钮到列表页
2. 调用现有的订单 API 接口
3. 使用项目标准的 Excel 导出工具

## 待确认项
- 导出字段范围需要确认
- 是否需要支持批量导出

## 进度追踪
- [ ] 已记录到 task_breakdown.md
```

### 示例 2：代码审查

**用户**：帮我审查这个组件是否符合项目规范

**技能执行过程**：
1. 参考 `project_guidelines.md` 的交付物标准
2. 检查代码是否能直接使用
3. 验证命名是否与现有文档一致
4. 确认是否可以映射到阶段验收项

**预期输出**：
```markdown
## 合规性评估

✅ 符合项：
- 使用了项目标准的 SCSS 命名规范
- 组件拆分合理
- 包含必要的类型定义

⚠️ 需要改进：
- 缺少错误处理逻辑
- 需要补充单元测试

## 建议修改
[具体的代码修改建议]
```

### 示例 3：流程咨询

**用户**：这个项目现在应该做什么？

**技能执行过程**：
1. 读取 `project_status.md` 获取阶段信息
2. 读取 `task_breakdown.md` 获取完成情况
3. 基于 `WORKFLOW.md` 说明下一步流程

**预期输出**：
```markdown
## 当前阶段
[M1/M2/M3] - 具体阶段描述

## 已完成
- 任务 A ✅
- 任务 B ✅

## 下一步
1. 任务 C（优先级：高）
2. 任务 D（优先级：中）

## 风险点
- 需要注意的边界条件
```

## 最佳实践

### 1. 提供清晰的任务描述

好的描述能让技能更准确地提供帮助：

✅ **好**："按照项目规范，帮我设计订单筛选功能的实现方案"
❌ **差**："做个订单筛选"

### 2. 明确约束条件

如果有特殊要求，提前说明：

✅ **好**："需要在 M1 阶段内完成，不要引入新的依赖库"
❌ **差**："尽快做完"

### 3. 反馈执行结果

使用后反馈是否有帮助：

✅ **好**："这个方案符合我们的规范，开始实施"
❌ **差**：（无反馈）

## 技能维护

### 更新时机

当出现以下情况时，应更新此技能：

1. 项目阶段发生重大变化
2. 工作流规范有重要调整
3. 发现技能未能正确响应的场景
4. 新增了必须遵循的规范文档

### 更新方法

1. 编辑 `.lingma/skills/lingma-workflow/SKILL.md`
2. 同步更新本文档（如有需要）
3. 在 `task_breakdown.md` 中记录变更

## 故障排查

### 问题：技能未被触发

**可能原因**：
- 用户请求中缺少触发关键词
- 技能描述不够明确

**解决方案**：
- 使用 `/lingma-workflow` 命令手动触发
- 在请求中明确提到"工作流"、"规范"等词

### 问题：响应不符合预期

**可能原因**：
- 工作流文档本身不清晰
- 项目阶段判断有误

**解决方案**：
- 检查工作流文档是否需要更新
- 在请求中提供更多上下文信息

## 与其他技能的协作

本技能可以与其他技能配合使用：

- **+ code-review**：在代码审查时同时遵循工作流规范
- **+ commit-helper**：生成符合项目规范的提交信息
- **+ doc-generator**：基于工作流模板生成文档
